# Архітектура та доменна логіка (лабораторна 5)

## Загальна схема

Додаток — **моноліт із тонким HTTP-шаром**: браузер або тести звертаються до **Express** (`server.js`), який читає/пише **SQLite** через **одне підключення** з `db.js`. Бізнес-правила для розрахунків і перевірок винесені в **`logic.js`**, щоб їх можна було тестувати окремо (unit-тести) та перевикористати в Cucumber-кроках.

```
[клієнт / supertest / HTML]
          │
          ▼
   server.js (REST)
          │
          ├─► logic.js (чисті функції: чек, VIN, номер)
          │
          └─► db.js → sto.db (SQLite)
```

## Архітектурні рішення

1. **Один файл `db.js`** створює таблиці при імпорті (у `serialize`) та експортує екземпляр `Database`. Це спрощує локальний запуск, але означає, що тести й сервер ділять один файл БД — для продакшену потрібні були б міграції та окремі БД для тестів.

2. **`server.js` експортує `app` без `listen` у режимі `NODE_ENV=test`**, щоб Jest міг викликати маршрути через `supertest` без відкритого порту.

3. **Розрахунок вартості замовлення** (`POST /api/orders`): спочатку підставляється SQL-фільтр за `items` (id послуг), підсумовуються ціни з `services`; якщо рядків немає — використовується **фіксована сума 500** (учбова логіка «дефолтної» вартості).

4. **Інвойс** (`GET /api/orders/:id/invoice`): податок моделюється як **5% від `total_cost`** замовлення без окремої таблиці податків.

## Бізнес-логіка (`logic.js`)

| Метод | Призначення |
|--------|-------------|
| `calculateInvoice` | ПДВ/податок 5%, захист від від'ємної суми. |
| `isValidVIN` | VIN саме **17 символів**, рядок. |
| `applyDiscount` | Знижка 10% при сумі **> 5000**. |
| `formatPlate` | Нормалізація держномера (без пробілів/дефісів, upper case). |
| `getStatusColor` | Мапінг статусу замовлення на колір для UI. |

Детальні контракти — у **JSDoc** у вихідному коді та у згенерованій документації (`npm run docs`).

## Взаємодія компонентів

- **Фронт** (`public/index.html`) викликає той самий REST, що й інтеграційні тести.
- **Unit-тести** (`tests/unit.test.js`) перевіряють лише `logic.js` — швидке регресійне покриття правил.
- **Інтеграційні тести** (`tests/integration.test.js`) піднімають `app` і перевіряють HTTP + реакцію БД.
- **Cucumber** описує сценарії мовою Gherkin; кроки викликають перевірки з `logic.js` (BDD поверх домену).

## Складні / неочевидні місця

- **SQL Injection у прайсі замовлення**: рядок `id IN (${items.join(',')})` без параметризації — **навмисно простий учбовий варіант**; у реальному проєкті потрібні плейсхолдери або білий список id.
- **`services` у двох варіантах схеми**: у `database.sql` поле може називатися `name`, у `db.js`/`seed.js` — `title`; узгоджуйте схему перед продакшеном.
