# Лінтинг і статичний аналіз (лабораторна 4)

## Дослідження інструментів

Для **JavaScript / Node.js / Express** найпоширеніші статичні аналізатори:

| Інструмент      | Призначення |
|-----------------|------------|
| **ESLint**      | Стиль, помилки, best practices, плагіни (наприклад, для Jest). |
| **Prettier**    | Автоформатування (не обов’язковий, часто поєднують з ESLint). |
| **TypeScript + `tsc`** | Статична типізація / перевірка `.js` через `checkJs`. |

Інші варіанти для екосистеми JS: **Semgrep**, **Sonar**, вбудовані перевірки в IDE — вони корисні для безпеки та складних правил, але **ESLint** залишається базовим стандартом для щоденної розробки.

## Які аспекти якості важливі для цього проєкту

1. **Стиль і узгодженість** — один стиль у `server.js`, тестах і Cucumber-кроках.  
2. **Помилки та «пастки»** — наприклад, умовні `expect` у Jest, зайві змінні.  
3. **Підтримуваність тестів** — правила `eslint-plugin-jest` для інтеграційних тестів.  
4. **Типова безпека типів** — базова перевірка через TypeScript `checkJs` без повної міграції на TS.

## Обраний лінтер

**ESLint** (плоска конфігурація `eslint.config.mjs`, ESLint 10):

- Нативна підтримка Node.js і тестів.  
- Легко ігнорувати `node_modules`, `coverage`, `public`, окремі скрипти.  
- Плагін **eslint-plugin-jest** для тестів Jest.

**Чому не лише Prettier:** Prettier не замінює семантичні правила (невикористані змінні, порівняння тощо).

## Базові правила та пояснення

| Правило | Режим | Навіщо |
|---------|--------|--------|
| `@eslint/js` recommended | error/warn за замовчуванням | Загальні проблеми JS. |
| `no-unused-vars` | warn, префікс `_` для навмисних ігнорів | Менше «сміття» в коді. |
| `eqeqeq: smart` | error | Уникаємо небезпечних `==`. |
| `no-console` | off | `seed.js` та утиліти використовують `console`. |
| `eslint-plugin-jest` recommended | у `*.test.js` | Коректні патерни тестів (наприклад, без умовних `expect`). |

## Ігнорування файлів і каталогів

У `eslint.config.mjs` в блоці `ignores` виключено:

- `node_modules/**` — залежності.  
- `coverage/**` — звіти Jest.  
- `public/**` — статична розмітка, не аналізується як Node-скрипт.  
- `load_test.js` — сценарій **k6** (інший рантайм / ESM), не сумісний із звичайною перевіркою CommonJS-проєкту.

## Як запускати

```bash
npm run lint          # перевірка всього проєкту
npm run lint:fix      # авто-виправлення, де можливо
npm run typecheck     # TypeScript: tsc --noEmit (checkJs)
npm run check         # lint + typecheck (швидка «якість коду»)
npm run build         # check + unit-тести (збірка/CI-подібний пайплайн)
```

## Звіт після першого запуску лінтера

Перша команда: `npx eslint .`

**Результат:** 7 порушень за категоріями ESLint: **2 error**, **5 warning** (сума враховується як одна множина «проблем» для відсотка).

**Що виправлено:**

1. `server.js` — невикористане поле `email` у деструктуризації (перейменовано на `_email`).  
2. `features/step_definitions/steps.js` — невикористаний параметр кроку (перейменовано на `_action`).  
3. `tests/integration.test.js` — `jest/no-conditional-expect`: перевірка полів клієнта через цикл `for … of` без умовних `expect`.  
4. `tests/logic.js` — невикористані змінні: додано `module.exports`.  
5. Додатково: синтаксис останнього тесту в `integration.test.js`, стійкий до конфлікту БД тест `IT-1` (унікальний телефон + коректне очікування з `db.get`).

## Як пораховано відсоток виправлень (50% / 90% / 100%)

- Позначимо **N** = загальна кількість порушень після **першого** повного `eslint .` (у нашому прогоні **N = 7**).  
- Позначимо **R** = скільки порушень залишилось після змін (**R = 0**).  
- **Частка усунутих** = \((N - R) / N\).

Приклади:

- **50%:** потрібно усунути щонайменше \(\lceil 0{,}5 \cdot N \rceil\) порушень. При **N = 7** це **4** з **7**.  
- **90%:** \(\lceil 0{,}9 \cdot N \rceil = 7\) з **7** — фактично всі.  
- **100%:** **R = 0** після `eslint .` (як зараз).

Перевірка після правок: `npm run lint` завершується без помилок і попереджень.

## Рекомендація щодо комітів (завдання 60%)

Зручно рознести так:

1. Коміт з інфраструктурою: ESLint, `tsconfig`, Husky, `docs/linting.md`, скрипти в `package.json`.  
2. Окремий коміт **лише з виправленнями коду** після `eslint` (мінімум 3 осмислені правки — у нас їх більше).

## Git hooks (завдання 100%)

- Встановлено **Husky** (`prepare`: `husky` у `package.json`).  
- Хук **`.husky/pre-commit`** викликає **`npx lint-staged`**.  
- **lint-staged** для кожного **заіндексованого** `*.js` запускає `eslint --max-warnings=0`, тобто коміт блокується, якщо у змінених JS-файлах є помилки лінтера.

## Інтеграція з процесом збірки

- Скрипт **`npm run build`** виконує: **`npm run check`** (lint + typecheck) **і** **`npm test`**.  
- Так лінтинг і типи входять у «фінальну» перевірку перед релізом / CI.

## Статична типізація

- Додано **TypeScript** як інструмент перевірки **без перетворення коду на TS**.  
- У **`tsconfig.json`**: `allowJs: true`, `checkJs: true`, `noEmit: true`, обмежений `include` серверних файлів, тестів і Cucumber-кроків.  
- Запуск: **`npm run typecheck`** (`tsc --noEmit`).

## Комплексна перевірка

Одна команда для якості коду перед здачею:

```bash
npm run check    # ESLint + tsc
# або повний цикл:
npm run build
```
