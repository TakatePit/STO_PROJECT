# Резервне копіювання (для release engineer / DevOps)

Проєкт **STO** зберігає дані у **файлі SQLite `sto.db`** у корені репозиторію (разом з можливими `sto.db-wal`, `sto.db-shm` у режимі WAL). Цей документ описує безпечне створення копій і перевірку відновлення.

Пов’язані документи: **`docs/deployment.md`**, **`docs/update.md`**.

## 1. Що саме копіювати

| Об’єкт | Примітка |
|--------|----------|
| **`sto.db`** | Основна база; обов’язково. |
| **`sto.db-wal`**, **`sto.db-shm`** | Якщо існують (WAL mode) — копіювати **разом** з `sto.db` у консистентний момент або використовувати метод нижче. |
| Код / `package-lock.json` | Зазвичай у Git; бекап коду не заміняє VCS, але може прискорити rollback. |

## 2. Коли робити бекап

- **Перед** кожним оновленням production (обов’язково) — див. **`docs/update.md`**.
- **За розкладом** (наприклад, щодня) для production з реальними даними.
- Після великих імпортів або масових змін даних — за політикою організації.

## 3. Спосіб A: зупинка додатка + копія файлів

Найпростіший для учбового однопроцесового деплою.

1. Зупинити Node (щоб не було активного запису):

   ```bash
   sudo systemctl stop sto
   ```

2. Створити копію з міткою часу:

   ```bash
   cd /opt/sto_project
   ts=$(date +%Y%m%d-%H%M%S)
   cp sto.db "sto.db.bak.$ts"
   cp sto.db-wal "sto.db-wal.bak.$ts" 2>/dev/null || true
   cp sto.db-shm "sto.db-shm.bak.$ts" 2>/dev/null || true
   ```

3. За бажанням упакувати в архів:

   ```bash
   tar czvf "sto-backup-$ts.tar.gz" sto.db sto.db-wal sto.db-shm 2>/dev/null || tar czvf "sto-backup-$ts.tar.gz" sto.db
   ```

4. Перенести архів на **інший диск** або **об’єктне сховище** (S3, інший сервер).

5. Запустити додаток знову: `sudo systemctl start sto`.

## 4. Спосіб B: онлайн-копія через SQLite (без довгої зупинки)

Якщо встановлено CLI `sqlite3`:

```bash
sqlite3 sto.db ".backup 'sto.db.backup-$(date +%Y%m%d-%H%M%S)'"
```

Переконайтеся, що файл бекапу записано повністю і **перевірте** його на копії:

```bash
sqlite3 sto.db.backup-XXXX "PRAGMA integrity_check;"
```

Очікуваний результат: `ok`.

## 5. Windows

1. Зупинити службу / процес Node (служби, NSSM або диспетчер завдань).  
2. Скопіювати `sto.db` у `sto.db.bak.%date%` або через PowerShell:

   ```powershell
   Copy-Item sto.db ("sto.db.bak." + (Get-Date -Format "yyyyMMdd-HHmmss"))
   ```

3. Зберігати копії **поза** сервером розробки, якщо це production.

## 6. Ротація та зберігання

- Тримати принаймні **останні N** поколінь (наприклад, 7 денних + 4 тижневих), залежно від політики.
- Не зберігати бекапи лише на тому самому диску, що й робоча БД.

## 7. Перевірка відновлення (тест)

На **окремій** машині або в тимчасовому каталозі:

1. Розпакувати копію `sto.db`.  
2. Запустити `sqlite3 sto.db "PRAGMA integrity_check;"` — має бути `ok`.  
3. За потреби підняти **копію** додатка з `NODE_ENV=production` на іншому порту і виконати `GET /api/clients`.

## 8. Відновлення після збою

Коротко: зупинити додаток → замінити `sto.db` (та WAL/SHM за потреби) на останню перевірену копію → виставити права доступу → запустити додаток. Деталізований **відкат** після невдалого оновлення — у **`docs/update.md`**, розділ rollback.
