# Оновлення системи та відкат (для release engineer / DevOps)

Цей документ описує обережне оновлення **STO Project** у production з **SQLite** та одним процесом Node.js. Адаптуйте кроки під ваш CI/CD (GitLab CI, GitHub Actions, Jenkins).

Початкове розгортання: **`docs/deployment.md`**.  
Резервні копії: **`docs/backup.md`**.

## A. Підготовка до оновлення

1. **Ознайомлення з release notes / комітами** — що змінилось у API, схемі БД, залежностях (`package.json`).
2. **Вікно обслуговування:** для учбового одноінстансового деплою простій можливий на **1–5 хв** під час перезапуску Node; узгодити з замовником за потреби.
3. **Перевірка сумісності:**
   - чи нова гілка/тег збирається: `npm ci && npm run build` на staging;
   - чи потрібні зміни в схемі БД (порівняти `db.js` / `database.sql` між версіями).

## B. Створення резервних копій перед оновленням

Обов’язково виконати **перед** зміною коду або міграціями. Детальна процедура: **`docs/backup.md`**.

Мінімум: знімок **`sto.db`** (і за наявності `sto.db-wal`, `sto.db-shm`) з timestamp у імені файлу після зупинки служби або через безпечну копію SQLite.

## C. Планування простою

- Якщо оновлення лише **код Node** без міграцій БД — простій = перезапуск процесу (**секунди**).
- Якщо потрібна **міграція SQLite** (рідко в цьому проєкті) — закласти час на бекап, міграцію, перевірку.

## D. Процес оновлення (покроково)

### 1. Зупинка служб

```bash
sudo systemctl stop sto
```

(Або `pm2 stop sto`, `docker stop …` — за вашим оркестратором.)

### 2. Розгортання нового коду

```bash
cd /opt/sto_project
git fetch --tags
git checkout <tag_або_гілка>
npm ci --omit=dev
```

Якщо деплой з **CI-артефакту** — розпакувати поверх робочого каталогу, не затерти `sto.db` без бекапу.

### 3. Міграція даних (якщо потрібно)

У поточному репозиторії таблиці створюються через **`db.js`** при старті (`CREATE TABLE IF NOT EXISTS`). Якщо з’являться **окремі SQL-скрипти міграцій**, виконати їх **до** старту нового коду або згідно з інструкцією релізу:

```bash
# Приклад (якщо буде tools/migrate-v2.sql):
# sqlite3 sto.db < tools/migrate-v2.sql
```

Перевірити цілісність: `sqlite3 sto.db "PRAGMA integrity_check;"`.

### 4. Оновлення конфігурацій

- Оновити **systemd** unit / **.env** (якщо з’являться) без коміту секретів у Git.
- Переконатися, що **NODE_ENV=production** для prod.

### 5. Запуск

```bash
sudo systemctl start sto
sudo systemctl status sto
```

### 6. Перевірка після оновлення

Ті самі кроки, що в **`docs/deployment.md`** (розділ «Перевірка працездатності»), плюс:

- один **критичний** бізнес-запит (наприклад, створення клієнта через API, якщо дозволено політикою).

Якщо щось не так — переходьте до **відкату**.

## E. Процедура відкату (rollback) при невдалому оновленні

1. **Зупинити** додаток:

   ```bash
   sudo systemctl stop sto
   ```

2. **Повернути код** на попередній тег/коміт:

   ```bash
   cd /opt/sto_project
   git checkout <попередній_тег>
   npm ci --omit=dev
   ```

3. **Повернути БД** з бекапу (якщо під час оновлення були зміни в даних або невдала міграція):

   ```bash
   cp sto.db.bak.XXXXXXXX sto.db
   chown sto:sto sto.db
   ```

4. **Запустити** знову:

   ```bash
   sudo systemctl start sto
   ```

5. **Перевірити** HTTP і API як у production-чеклісті в **`docs/deployment.md`**.

6. Зафіксувати інцидент: версія, симптоми, чи спрацював rollback повністю.

> **Примітка:** якщо новий реліз **не змінював** схему БД і дані залишились валідними, інколи достатньо відкотити лише **код** без заміни `sto.db`. Рішення приймає відповідальний за реліз після аналізу.
