# Розгортання у production (для release engineer / DevOps)

Документ описує типове розгортання учбового **STO Project** на одній Linux- або Windows-серверній ВМ з **Node.js** і файлом **SQLite**. Схема не високодоступна (single-instance); для курсової/лабораторної цього достатньо.

Короткий опис компонентів і діаграма: **`docs/deployment-architecture.md`**.  
Резервне копіювання: **`docs/backup.md`**.  
Оновлення та відкат: **`docs/update.md`**.

## 1. Вимоги до апаратного забезпечення

| Ресурс | Мінімум (демо) | Рекомендовано (мала продакшен-навантаження) |
|--------|------------------|---------------------------------------------|
| **Архітектура CPU** | x86_64 або ARM64 | x86_64 |
| **vCPU** | 1 | 2+ |
| **RAM** | 512 MiB | 2 GiB (запас під Node + ОС) |
| **Диск** | 2 GB вільних | 10 GB+ SSD (логи, резервні копії `sto.db`) |
| **Мережа** | Порт 3000 або 80/443 через проксі | Стабільний вихід у інтернет для `npm ci` на етапі збірки |

## 2. Необхідне програмне забезпечення

- **ОС:** Ubuntu Server 22.04 LTS (або новіша), Windows Server / Windows 10+ — за політикою організації.
- **Node.js:** **LTS** (наприклад, 20.x або 22.x), 64-bit. Перевірка: `node -v`, `npm -v`.
- **Git:** для отримання коду з репозиторію.
- **Опційно (рекомендовано для production):**
  - **Nginx** або **Caddy** — TLS, gzip, зворотний проксі на `127.0.0.1:3000`.
  - **systemd** (Linux) або **NSSM** / планувальник завдань (Windows) — керування процесом Node.
  - **PM2** або **Foreman** — альтернатива, див. `Procfile` у корені репозиторію.

Залежності проєкту встановлюються з **`package.json`**; для reproducible build у prod краще **`npm ci`** (потрібен `package-lock.json`).

## 3. Налаштування мережі

1. Відкрити вхідний порт лише за потреби:
   - або **прямо** TCP **3000** (внутрішня мережа / dev);
   - або **80/443** на **зворотному проксі**, який проксує на `http://127.0.0.1:3000`.
2. **Firewall** (ufw, security group у хмарі): дозволити SSH з дозволених IP; HTTP/HTTPS — на проксі або на Node (не рекомендовано без TLS).
3. Якщо додаток за NAT — налаштувати **port forwarding** або LB у хмарі.

## 4. Конфігурація сервера (приклад Linux + systemd)

1. Створити системного користувача `sto` (опційно), каталог `/opt/sto_project`.
2. Клонувати репозиторій у `/opt/sto_project` (або розгорнути артефакт CI).
3. У каталозі проєкту:

   ```bash
   npm ci --omit=dev
   ```

   Для повної збірки з тестами на окремому build-агенті використовуйте `npm ci` і `npm run build`, потім на сервері — лише production-залежності, якщо прийнята така політика.

4. Змінні середовища (наприклад, у юніті systemd):

   - `NODE_ENV=production`
   - За потреби `PORT=3000` (якщо додасте читання порту з env у коді — наразі порт захардкожено в `server.js`; для зміни потрібна правка або проксі).

5. Юніт `systemd` (ескіз `/etc/systemd/system/sto.service`):

   ```ini
   [Unit]
   Description=STO ERP Node
   After=network.target

   [Service]
   Type=simple
   User=sto
   WorkingDirectory=/opt/sto_project
   Environment=NODE_ENV=production
   ExecStart=/usr/bin/node server.js
   Restart=on-failure

   [Install]
   WantedBy=multi-user.target
   ```

   ```bash
   sudo systemctl daemon-reload
   sudo systemctl enable --now sto
   ```

## 5. Налаштування СУБД (SQLite)

- Файл **`sto.db`** створюється/оновлюється при старті через **`db.js`** (таблиці `CREATE TABLE IF NOT EXISTS`).
- **Розміщення:** у корені проєкту поряд з `server.js` (шлях `./sto.db` у коді). Для production зручно задати **абсолютний шлях** через змінну середовища в майбутніх версіях; зараз — переконатися, що `WorkingDirectory` процесу = корінь проєкту.
- **Права:** процес Node має мати **читання й запис** до `sto.db` і каталогу (SQLite може створювати `-wal`, `-shm` файли).
- **Резервне копіювання:** див. **`docs/backup.md`**.

Якщо застосовується **`database.sql`** / **`seed.js`** — запускати **одноразово** або в controlled release (не на кожен старт, якщо не потрібне скидання даних).

## 6. Скрипти та конфігурації в `docs/scripts`

| Файл | Опис |
|------|------|
| `docs/scripts/dev.bat` / `docs/scripts/dev.sh` | `npm install` з кореня репозиторію (онбординг). |
| `docs/scripts/start-prod.bat` / `docs/scripts/start-prod.sh` | Запуск `node` із **`NODE_ENV=production`** (робочий каталог — корінь репо). |
| `docs/scripts/run-prod.js` | Використовується **`npm run start:prod`**. |
| `Procfile` (корінь) | Процес `web` для **Foreman** (`foreman start`). |

## 7. Розгортання коду (узагальнений pipeline)

1. Отримати версію (git tag / artifact).  
2. `npm ci --omit=dev` на сервері (або скопіювати вже зібраний `node_modules` з довіреного артефакту).  
3. Переконатися, що **`sto.db`** або новий порожній файл на місці з коректними правами.  
4. Перезапустити сервіс: `systemctl restart sto` (або еквівалент).  
5. Виконати **перевірку працездатності** (нижче).  
6. Перед релізом за потреби виконати **бекап** згідно **`docs/backup.md`**.

## 8. Перевірка працездатності

| Крок | Дія | Очікуваний результат |
|------|-----|----------------------|
| 1 | `curl -sSf http://127.0.0.1:3000/` або відкрити в браузері головну | HTML сторінка з ERP |
| 2 | `curl -sSf http://127.0.0.1:3000/api/clients` | JSON-масив (може бути `[]`) |
| 3 | Перевірити логи процесу (`journalctl -u sto -f`) | Немає необроблених crash; стартове повідомлення про порт |
| 4 | (Опційно) smoke-тест з іншої машини | HTTP 200 через проксі |

Якщо використовується Nginx — перевірити також `https://.../api/clients` через проксі.

## 9. Додаткові зауваження
