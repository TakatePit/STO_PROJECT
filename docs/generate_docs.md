# Генерація документації з коду (JSDoc)

У проєкті використовується **[JSDoc 3](https://jsdoc.app/)** — стандарт для JavaScript/Node.js: коментарі `/** ... */` біля модулів, функцій, типів; з них будується статичний HTML-сайт.

Для опису REST API на рівні HTTP також поширені **OpenAPI (Swagger)** і генерація з анотацій або окремого `openapi.yaml`; у цьому навчальному репозиторії обрано JSDoc як найпростіший шлях без додаткового рантайму.

## Вимоги

- Встановлені залежності: `npm install`
- Згенерований каталог **`docs/jsdoc/`** (див. нижче) **не комітиться** (у `.gitignore`), щоб не засмічувати репозиторій мініфікованими скриптами JSDoc.

## Запуск генерації

З кореня репозиторію:

```bash
npm run docs
```

Конфігурація: **`jsdoc.json`**  
Вихід: каталог **`docs/jsdoc/`** (головна сторінка: `docs/jsdoc/index.html`).

Відкрий `index.html` у браузері або підніми локально:

```bash
npx http-server docs/jsdoc -p 8080
```

## Перевірка якості коментарів (лінтер)

Правила **eslint-plugin-jsdoc** застосовуються до файлів:

`logic.js`, `db.js`, `server.js`, `seed.js`, `tests/logic.js`

загалом через:

```bash
npm run lint
```

Після змін у JSDoc слід прогнати `npm run lint` і виправити попередження.

## Які файли потрапляють у JSDoc

У `jsdoc.json` поле `source.include` обмежує набір:

- `server.js` — REST API  
- `db.js` — SQLite  
- `logic.js` — бізнес-логіка  
- `seed.js` — наповнення БД  
- `tests/logic.js` — допоміжні тестові функції з документованими контрактами  

Cucumber-кроки та самі тестові сценарії в JSDoc не включені; їх роль описана в **`docs/architecture.md`** і в розділі README про «живу» документацію.

## Архів для здачі завдання

1. Виконай `npm run docs`.  
2. У провіднику Windows: клацни правою по папці **`docs/jsdoc`** → **Надіслати до** → **Стисла папка (zip)** (або `Compress-Archive` у PowerShell).  
3. Додай отриманий архів до системи здачі лабораторної.

Приклад у PowerShell із кореня проєкту:

```powershell
npm run docs
Compress-Archive -Path docs\jsdoc -DestinationPath sto-jsdoc.zip -Force
```

## Оновлення документації

- Будь-яка зміна сигнатури або поведінки публічного API → онови JSDoc у відповідному файлі.  
- Після змін у доменних правилах оновлюй також **`docs/architecture.md`** (якщо змінилась схема або алгоритми).  
- Приклади в **`tests/unit.test.js`** та **`tests/integration.test.js`** мають узгоджуватись із JSDoc.
